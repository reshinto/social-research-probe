# Project Structure

A file-by-file and folder-by-folder reference for the `social-research-probe` repository. Use this when you need to find where something lives, understand what a module owns, or decide where to add new code.

---

## 1. Root files

### `README.md`
The public-facing project overview. Covers what `srp` does, quickstart, installation, usage examples, the documentation table, architecture summary, and contribution notes. Update it when major features are added or the CLI interface changes.

### `CHANGELOG.md`
Release history in reverse chronological order. The GitHub release workflow uses this as its release body. Add an entry for every user-visible change before bumping `VERSION`.

### `CONTRIBUTING.md`
Developer workflow rules: TDD requirements, 100% branch coverage gate, file-size limits (500 lines), commit conventions, and how to run lint/tests locally. Read this before opening a pull request.

### `CODE_OF_CONDUCT.md`
Community standards for the project. Rarely modified; update only if the policy changes.

### `SECURITY.md`
Responsible disclosure policy and contact instructions for reporting vulnerabilities. Modify only if the security contact or process changes.

### `LICENSE`
MIT license text. Do not modify.

### `VERSION`
A single line containing the current semver string (e.g. `0.0.1`). Hatchling reads this file to set the package version. The `release.yml` workflow triggers on a push to `main` that changes this file. Bump this file to cut a release.

### `pyproject.toml`
The single source of build truth. Key sections:
- `[build-system]` â€” uses Hatchling; version is read from `VERSION`.
- `[project.dependencies]` â€” runtime deps: `jsonschema`, `rapidfuzz`, `google-api-python-client`, `yt-dlp`, `youtube-transcript-api`, `matplotlib`, `openai-whisper`, `httpx`.
- `[project.scripts]` â€” registers the `srp` entry point.
- `[tool.pytest.ini_options]` â€” enforces 100% branch coverage via `--cov-fail-under=100`; `asyncio_mode = "auto"`.
- `[tool.coverage.report]` â€” `exclude_lines` for abstract/protocol boilerplate.
- `[tool.ruff]` â€” lint rules (E, F, I, N, UP, B, SIM, RUF); line length 100; Python 3.11 target.

### `requirements.txt`
Pinned runtime dependencies mirroring `[project.dependencies]`. Keep in sync with `pyproject.toml` whenever dependencies change.

### `requirements-dev.txt`
Starts with `-r requirements.txt`, then adds dev-only tools: `pytest`, `pytest-cov`, `pytest-asyncio`, `hypothesis`, `ruff`, `respx`, `vulture`. Keep in sync with `pyproject.toml` `[project.optional-dependencies]` and `[dependency-groups]`.

### `uv.lock`
Lockfile generated by `uv`. Do not hand-edit; regenerate with `uv lock` after changing dependencies.

### `.env.example`
Documents every environment variable the tool reads (e.g. `YOUTUBE_API_KEY`, `EXA_API_KEY`, `BRAVE_API_KEY`, `TAVILY_API_KEY`). The real secrets live in `~/.social-research-probe/secrets.toml`. Consult this file when configuring a new environment.

### `config.toml.example`
Annotated example of `~/.social-research-probe/config.toml`. Documents all configurable keys: LLM runner selection, timeout, per-runner model flags, scoring weight overrides, corroboration backend, and YouTube adapter defaults. Copy to your data directory and edit as needed.

### `install.sh`
Convenience shell script for first-time setup: installs the package, sets up the data directory, and walks through initial configuration. Useful for CI environments and onboarding.

### `.gitignore`
Standard Python ignores plus project-specific entries: the `.venv/` directory, `__pycache__/`, `dist/`, `*.egg-info`, `*.pyc`, and any local `config.toml` / `secrets.toml` files.

## 3. `social_research_probe/` â€” the main package

### Root-level modules

**`__init__.py`**
Package init. Exposes the package version and re-exports the public surface.

**`__main__.py`**
Allows `python -m social_research_probe` to invoke the CLI. Covered by a coverage exclusion so it does not inflate branch counts.

**`types.py`**
Single vocabulary for every major typed payload in the system. Defines `TypedDict` classes for `AppConfig`, `AdapterConfig`, `ResearchPacket`, `MultiResearchPacket`, `ScoredItem`, `PurposeEntry`, `TopicsState`, `PurposesState`, `PendingSuggestionsState`, scoring breakdowns, and more. Import from here instead of inline `dict` annotations anywhere in the codebase.

**`packet.py`**
Defines the `PacketEnvelope` wrapper and `wrap_packet` / `unwrap_packet` helpers. Every CLI command that emits a result uses this envelope so callers can distinguish `synthesis`, `suggestions`, and `corroboration` payloads.

**`config.py`**
Loads and merges `config.toml` with built-in defaults, resolves the data directory (from `--data-dir` flag, `SRP_DATA_DIR` env var, or `~/.social-research-probe/`), and exposes a typed `Config` dataclass. Owns non-secret runtime configuration; secrets stay in `commands/config.py`.

**`dedupe.py`**
Fuzzy duplicate detection using `rapidfuzz` `token_set_ratio`. Provides `DuplicateStatus` enum (`new`, `near-duplicate`, `duplicate`) and `DedupeResult`. Used when staging topic/purpose suggestions to prevent near-identical entries.

**`errors.py`**
Exception hierarchy keyed to spec Â§9 exit codes: `SrpError` (2), `ValidationError` (2), `DuplicateError` (3), `AdapterError` (4), `SynthesisError` (4), `MigrationError` (5). Raise the most specific subclass so the CLI can emit the correct exit code.

---

### `cli/`

Entry point and argument dispatch layer. Separates argument parsing from handler logic.

**`__init__.py`** / **`__main__.py`**
`main()` entry point wired to the `srp` console script. Calls `_global_parser()`, resolves the data directory, routes to the appropriate handler, and handles `SrpError` exit codes.

**`parsers.py`**
Builds the root `argparse` parser and registers every subcommand: `update-topics`, `show-topics`, `update-purposes`, `show-purposes`, `suggest-topics`, `suggest-purposes`, `show-pending`, `apply-pending`, `discard-pending`, `stage-suggestions`, `research`, `corroborate-claims`, `render`, `install-skill`, `report`, and `config`. Modify here when adding or changing CLI flags.

**`handlers.py`**
One `_handle_*` function per subcommand. Calls into `commands/` modules and emits results via `_emit`. Imports command modules lazily inside handlers to keep startup fast. Modify here when changing handler logic or wiring a new subcommand.

**`utils.py`**
Small private helpers shared between `handlers.py` and the main entry point: `_emit` (formats output as text/JSON/Markdown), `_id_selector` (parses comma-separated ID lists for pending suggestions).

---

### `commands/`

One module per CLI subcommand. These contain the domain logic that handlers delegate to.

**`config.py`**
Reads, writes, and validates `config.toml` and `secrets.toml`. Implements `show_config`, `write_config_value`, `write_secret`, `unset_secret`, and `check_secrets`. The secrets file must be `chmod 0600`.

**`topics.py`**
CRUD operations on the topics list persisted in `state/topics.json`: `add_topics`, `remove_topics`, `rename_topic`, `show_topics`. Enforces deduplication via `dedupe.py`.

**`purposes.py`**
CRUD operations on the purposes map in `state/purposes.json`: `add_purpose`, `remove_purposes`, `rename_purpose`, `show_purposes`. Each purpose has a `method` string, `evidence_priorities`, and `scoring_overrides`.

**`suggestions.py`**
Generates, stages, applies, and discards topic/purpose suggestions. `suggest_topics` and `suggest_purposes` use heuristics or an LLM runner; `stage_suggestions` writes to `state/pending_suggestions.json` after deduplication; `apply_pending` / `discard_pending` act on staged suggestions by ID.

**`parse.py`**
Argument parsing helpers used by handlers: `_parse_quoted_list`, `_take_quoted`, and `ParsedRunResearch` (dataclass holding the resolved topic, platform, and purpose list from `research` subcommand positional args).

**`nl_query.py`**
Natural-language query classifier. When `srp research` receives a single free-text argument, this module uses the configured LLM runner to extract a structured `(topic, purposes)` pair, enabling queries like `srp research "who is winning the LLM benchmarks?"`.

**`corroborate_claims.py`**
Standalone claim corroboration command. Reads a claims JSON file, calls the requested backends, and writes the result to stdout or a file.

**`render.py`**
Renders charts and stats for an existing research packet JSON file without re-running the pipeline. Useful for regenerating visuals after configuration changes.

**`report.py`**
Re-renders the HTML report from a saved packet file, optionally overriding synthesis sections 10â€“11 from text files. Useful for offline report generation or custom synthesis injection.

**`install_skill.py`**
Copies `skill/SKILL.md` and `skill/references/` into `~/.claude/skills/srp/` (or a custom target), enabling the `/srp` Claude Code skill.

### `pipeline/`

Five-stage research orchestrator. Called by the `research` command handler.

**`orchestrator.py`**
Top-level `run_research()` function. Wires the five stages: fetch â†’ score â†’ enrich â†’ stats/charts â†’ synthesise. Loops over topics, assembles `ResearchPacket` objects, and produces a `MultiResearchPacket` when multiple topics are requested. Also resolves available corroboration backends from config.

**`enrichment.py`**
Stage 3: fetches transcripts and LLM summaries for the top-5 scored items. Calls the YouTube transcript pipeline and the LLM runner to generate `one_line_takeaway` strings.

**`corroboration.py`**
Stage 5 (claim checking): invokes configured corroboration backends against the top items, collecting verdict counts and evidence notes.

**`scoring.py`**
Helpers for stage 2: `_score_item`, `_enrich_query`, and `_zscore`. Computes per-item `ScoreBreakdown` using trust/trend/opportunity sub-scorers and normalises across the result set.

**`stats.py`**
Stage 4 (statistical analysis): calls the `stats/selector.py` to run the applicable statistical models for the current result set and returns a `StatsSummary`.

**`charts.py`**
Stage 4 (visualisation): calls `viz/selector.py` to render charts as PNGs and returns caption strings for inclusion in the HTML report.

**`svs.py`**
Builds the source-validation summary (`SourceValidationSummary`) by aggregating trust-hint classifications across all items.

---

### `platforms/`

Platform adapter layer. The pipeline always talks through `platforms/base.py` interfaces, never directly to platform APIs.

**`base.py`**
`PlatformAdapter` abstract base class defining the `fetch(topic, config)` coroutine contract, plus `RawItem`, `TrustHints`, and `FetchLimits` dataclasses.

**`registry.py`**
Maps platform name strings (`"youtube"`) to adapter instances. Call `get_adapter("youtube")` to retrieve the adapter. Register new platforms here.

**`youtube/adapter.py`**
Main YouTube adapter. Calls `fetch.py` to query the YouTube Data API and assembles `RawItem` results with trust hints.

**`youtube/fetch.py`**
Low-level YouTube Data API calls (search and channel detail). Applies the configurable `recency_days`, `max_items`, and TTL-based response caching.

**`youtube/extract.py`**
Extracts structured fields (views, likes, duration, subscriber count) from raw YouTube API response objects.

**`youtube/trust_hints.py`**
Derives `TrustHints` from channel metadata: subscriber thresholds, verified status, known-domain classification.

**`youtube/whisper_transcript.py`**
Fallback transcript fetcher: downloads audio via `yt-dlp` and transcribes with OpenAI Whisper when YouTube captions are unavailable.

---

### `corroboration/`

Claim corroboration backends. Each backend checks a claim against external evidence and returns a verdict.

**`base.py`**
`CorroborationBackend` abstract base class and `CorroborationResult` dataclass. All backends must implement `check(claim)` and `health_check()`.

**`registry.py`**
Maps backend name strings to backend instances. `get_backend(name)` is the single lookup point. Register new backends here.

**`host.py`**
Auto-discovery backend: tries `exa`, `brave`, and `tavily` in order and uses the first that passes `health_check()`.

**`exa.py`**, **`brave.py`**, **`tavily.py`**
Individual search-API backends. Each reads its API key from `secrets.toml` via `_secret_utils.py`, issues an HTTP search query via `httpx`, and returns a structured `CorroborationResult`.

**`llm_cli.py`**
LLM-based backend: prompts the configured LLM runner to assess a claim and returns a verdict. Used when no web-search key is available.

**`_secret_utils.py`**
Shared helper for loading API keys from `secrets.toml` and injecting the `HTTP_USER_AGENT` header constant.

---

### `llm/`

LLM runner registry and ensemble. The pipeline uses this layer; it never shells out to an LLM directly.

**`base.py`**
`LLMRunner` abstract base class: `run(prompt)` coroutine, `health_check()`, and runner name property.

**`registry.py`**
Maps runner name strings to runner instances. `get_runner(name)` is the single lookup point.

**`ensemble.py`**
Multi-LLM ensemble runner. Dispatches the same prompt to multiple runners in parallel via `ThreadPoolExecutor`, collects responses, and returns a merged result. Used for transcript summarisation.

**`host.py`**
Config-aware runner: reads `llm.runner` from `config.toml` and returns the matching registered runner.

**`prompts.py`**
Prompt templates for all LLM calls: transcript summarisation, NL query classification, topic/purpose suggestion generation, and synthesis.

**`schemas.py`**
JSON schemas for structured LLM outputs (synthesis JSON, suggestion candidates). Used for response validation.

**`runners/`**
One file per supported LLM: `claude.py` (Claude via `claude` CLI), `gemini.py` (Gemini via `gemini` CLI), `codex.py` (OpenAI Codex via `codex` CLI), `local.py` (local models via `ollama`). All extend `cli_json_base.py`, which handles subprocess invocation, timeout, and JSON parsing.

---

### `scoring/`

Trust, trend, and opportunity scorers. Each scorer is independent and operates on `ItemFeatures`.

**`trust.py`**
Computes a [0, 1] trust score from channel credibility signals: subscriber count, verified status, source classification, and corroboration verdict.

**`trend.py`**
Computes a [0, 1] trend score from temporal signals: view velocity (views per day since publish) and engagement ratio.

**`opportunity.py`**
Computes a [0, 1] opportunity score reflecting underexplored angles: low subscriber count relative to view count, niche topic signals.

**`combine.py`**
`overall_score(trust, trend, opportunity)` â€” weighted combination (0.45/0.30/0.25) clamped to [0, 1].

---

### `stats/`

Twenty statistical models, each in its own file. One file per statistical model; see [docs/statistics.md](statistics.md) for a full model reference. `selector.py` chooses applicable models based on dataset size and purpose, and `base.py` defines the `StatModel` abstract base class.

---

### `viz/`

Chart renderers. One file per chart type; see [docs/charts.md](charts.md) for details on each chart. `selector.py` decides which charts to render for a given packet. `base.py` defines the `ChartRenderer` abstract base. `_png_writer.py` handles file persistence. `ascii.py` produces terminal-friendly ASCII fallbacks.

---

### `synthesize/`

LLM-driven narrative assembly for the research packet.

**`evidence.py`**
`summarize()` builds the `evidence_summary` field: a short prose narrative of what the fetched items collectively show. `summarize_signals()` generates `platform_signals_summary`.

**`explain.py`**
Generates per-item `one_line_takeaway` strings by calling the LLM runner with a summarisation prompt.

**`explanations/`**
Prompt fragments and supporting files for explanation generation.

**`formatter.py`**
`build_packet()` assembles the full `ResearchPacket` from enriched items, stats, chart captions, and synthesis strings. Also owns `render_full()`, which formats the packet as an 11-section Markdown string for terminal output.

**`warnings.py`**
`detect()` scans the assembled packet for known quality issues (low item count, no transcripts, corroboration failures) and returns warning strings.

**`llm_contract.py`**
Validates that structured LLM synthesis outputs conform to `schemas.py` before they are attached to the packet.

---

### `render/`

HTML and Markdown report generation.

**`html.py`**
Renders a `ResearchPacket` as a self-contained HTML file. The report slug is derived from the topic (lowercased, non-word chars to hyphens, truncated at 40 chars). Sections 10â€“11 show placeholder text when no LLM synthesis is present.

**`_sections.py`**
HTML section builder functions (one per report section). Separates section markup from the top-level renderer.

**`_assets.py`**
Inline CSS, JavaScript, and base64-encoded image helpers for the self-contained HTML report.

**`markdown_to_html.py`**
Lightweight Markdown-to-HTML converter for report prose sections.

**`tts/http.py`**
Text-to-speech HTTP client used to generate audio summaries for the HTML report when a TTS endpoint is configured.

---

### `state/`

On-disk state for topics, purposes, and pending suggestions.

**`store.py`**
Atomic JSON reader/writer. Uses POSIX-atomic replace (write to temp file, `fsync`, then `rename`) to prevent corrupt state files. All state reads and writes go through this module.

**`schemas.py`**
JSON Schema definitions and default values for `topics.json`, `purposes.json`, and `pending_suggestions.json`. Used for validation on load.

**`migrate.py`**
Schema migration logic. `migrate_to_current()` upgrades state documents from older `schema_version` values to the current schema.

**`validate.py`**
Validates a loaded state document against its JSON Schema using `jsonschema`.

---

### `validation/`

Source and claim quality checks.

**`source.py`**
Classifies items into `primary`, `secondary`, or `commentary` source tiers based on URL domain patterns and channel metadata.

**`claims.py`**
Validates that a claims JSON file has the expected shape before passing it to corroboration backends.

**`ai_slop.py`**
Heuristic detector for low-quality AI-generated content signals in transcripts and titles.

---

### `utils/`

Shared infrastructure utilities with no domain logic.

**`cache.py`**
Simple TTL-based disk cache for platform API responses. Keyed by a hash of the query parameters.

**`hashing.py`**
Deterministic hash helpers (SHA-256 based) used by the cache and deduplication layers.

**`io.py`**
File I/O helpers: safe directory creation, atomic file writes, and path resolution utilities.

**`progress.py`**
Logs `[srp]` diagnostic messages to `stderr` so they do not pollute the JSON packet on `stdout`.

**`subprocess_runner.py`**
Runs an external command with a configurable timeout and captures stdout/stderr. Used by all LLM CLI runners.

---

### `purposes/`

Purpose registry and scoring-override merging.

**`registry.py`**
Loads `purposes.json` and provides typed lookup by purpose name. Returns `PurposeEntry` objects with `method`, `evidence_priorities`, and `scoring_overrides`.

**`merge.py`**
`merge_purposes()` merges per-purpose `scoring_overrides` with spec Â§6 global defaults using element-wise max.

---

### `skill/`

Claude Code skill bundle installed by `srp install-skill`.

**`SKILL.md`**
Skill manifest consumed by Claude Code. Declares the `srp` skill name, disables direct model invocation (Claude must shell out to `srp`), and points to `references/index.md` for the command map.

**`references/`**
Reference documents that the Claude Code skill reads to understand how to invoke each `srp` subcommand correctly. Includes `research.md` with the NL query form and full flag reference.

## 4. `tests/` â€” test organisation

**`conftest.py`**
Shared pytest fixtures (data-dir setup, fake platform registration, monkeypatching helpers). Every test tier imports from here implicitly.

**`fixtures/`**
Test double implementations:
- `fake_youtube.py` â€” a `PlatformAdapter` that returns deterministic canned items; activated by `SRP_TEST_USE_FAKE_YOUTUBE=1`.
- `fake_corroboration.py` â€” a `CorroborationBackend` that always returns a fixed verdict without network calls.

**`unit/`**
Fast, isolated unit tests. One test file per production module. Use `monkeypatch`, `fake_youtube`, and `fake_corroboration` to avoid all I/O. Must maintain 100% branch coverage across the `social_research_probe` package.

**`integration/`**
Slower tests that exercise real I/O paths or multi-module interactions. Run separately in CI (`pytest tests/integration --no-cov`) to avoid inflating the coverage gate. Includes end-to-end pipeline tests and state-CLI round-trips.

**`contract/`**
Structural assertions about the codebase itself:
- `test_docs_navigation.py` â€” every `.md` under `docs/` must be reachable from `docs/README.md` and carry a breadcrumb.
- `test_docs_diagrams.py` â€” diagram source files must resolve.
- `test_no_any_types.py` â€” prohibits bare `Any` annotations.
- `test_no_llm_sdk.py` â€” ensures no direct LLM SDK imports (all LLM calls go through `llm/runners/`).
- `test_version_consistency.py` â€” `VERSION` file and package metadata must agree.

**Conventions:**
- Test functions are named `test_<thing>_<scenario>`.
- `N802` (function-name case) is ignored in test files to allow `test_MyClass_*` patterns.
- All new files must reach 100% branch coverage before merging.

## 6. `.github/` and `scripts/`

### `.github/workflows/ci.yml`
Runs on every push to `main` and on all pull requests. Matrix: Python 3.11, 3.12, 3.13. Steps: install, `ruff check`, `ruff format --check`, integration tests (no coverage), unit + contract tests with 100% branch coverage gate.

### `.github/workflows/release.yml`
Triggered when `VERSION` is changed on a push to `main`. Steps: reads the new version, creates and pushes an annotated git tag, builds wheel and sdist with `python -m build`, computes SHA-256 checksums, creates a GitHub release with `CHANGELOG.md` as the body, and publishes to PyPI using OIDC trusted publishing.

### `scripts/tag_release.sh`
Idempotent local release helper. Reads `VERSION`, creates an annotated tag `v<version>`, and pushes it. Accepts `--yes` to skip the confirmation prompt and `--skip-existing` to exit cleanly if the tag already exists. Use this for manual releases; the automated path is `release.yml`.
